# Бизнес-требования к программе "Журнал личных учебных достижений"

## 1. Общие сведения

### 1.1 Наименование проекта
**Журнал личных учебных достижений** (Personal Academic Achievement Journal)

### 1.2 Цель проекта
Разработка десктопного приложения для систематизации, хранения и управления информацией о личных учебных достижениях студентов и учащихся.

### 1.3 Целевая аудитория
- Студенты вузов и колледжей
- Учащиеся школ и лицеев
- Преподаватели (для ведения портфолио студентов)
- Абитуриенты (для формирования портфолио достижений)
- Слушатели курсов повышения квалификации

## 2. Бизнес-потребности

### 2.1 Проблемы текущей ситуации
1. **Разрозненное хранение информации** - достижения фиксируются в разных форматах (бумажные сертификаты, электронные файлы, заметки)
2. **Сложность систематизации** - отсутствие единой структуры для классификации достижений
3. **Трудоемкость отчетности** - ручное формирование отчетов для портфолио, заявок на стипендии, резюме
4. **Риск потери данных** - бумажные документы могут быть утеряны, электронные файлы не имеют резервных копий
5. **Неэффективное использование времени** - поиск и систематизация информации занимает значительное время

### 2.2 Бизнес-цели
1. **Централизация данных** - создание единого репозитория для всех учебных достижений
2. **Стандартизация** - унификация формата хранения информации об достижениях
3. **Автоматизация** - снижение ручного труда при формировании отчетов
4. **Повышение эффективности** - сокращение времени на поиск и систематизацию информации на 70%
5. **Поддержка принятия решений** - предоставление аналитической информации для планирования учебного развития

### 2.3 Ключевые показатели успеха (KPI)
1. Увеличение полноты учета достижений на 90%
2. Сокращение времени на формирование портфолио с 4 часов до 30 минут
3. Увеличение количества оформляемых заявок на стипендии на 40%
4. Снижение потерь информации об достижениях до 0%
5. Удовлетворенность пользователей не менее 85%

## 3. Функциональные требования

### 3.1 Управление достижениями
**Требование 3.1.1:** Система должна позволять добавлять новые достижения с обязательными полями:
- Название достижения
- Дата получения (в формате ГГГГ-ММ-ДД)
- Тип достижения (из предустановленного списка)
- Уровень достижения (локальный, региональный, национальный, международный)
- Описание (опционально)

**Требование 3.1.2:** Система должна предоставлять возможность просмотра списка всех достижений с сортировкой по дате (от новых к старым).

**Требование 3.1.3:** Система должна позволять удалять достижения из базы данных с подтверждением действия.

### 3.2 Гибкая настройка типов
**Требование 3.2.1:** Система должна загружать типы достижений из внешнего JSON-файла, что позволяет администратору/пользователю настраивать классификацию без изменения кода программы.

**Требование 3.2.2:** При отсутствии или ошибке в файле конфигурации система должна использовать предустановленный список типов достижений.

### 3.3 Экспорт данных
**Требование 3.3.1:** Система должна предоставлять функцию экспорта всех достижений в формат Microsoft Word (.docx) с сохранением форматирования.

**Требование 3.3.2:** Экспортированный документ должен содержать:
- Заголовок отчета
- Дата формирования отчета
- Нумерованный список всех достижений с форматированием:
  - Название - жирный шрифт
  - Дата - курсив
  - Тип и уровень - обычный текст
  - Описание (если есть) - отдельным абзацем

### 3.4 Производительность
**Требование 3.4.1:** Время отклика системы на любую операцию (добавление, удаление, экспорт) не должно превышать 3 секунд при хранении до 1000 записей.

**Требование 3.4.2:** Программа должна корректно работать на компьютерах со следующими минимальными характеристиками:
- Процессор: 1 ГГц
- ОЗУ: 2 ГБ
- Свободное место на диске: 100 МБ

## 4. Нефункциональные требования

### 4.1 Требования к ПО
**Требование 4.1.1:** Приложение должно быть разработано на Python версии 3.8 или выше.

**Требование 4.1.2:** Для графического интерфейса должна использоваться только стандартная библиотека tkinter.

**Требование 4.1.3:** Для хранения данных должна использоваться встроенная база данных SQLite.

**Требование 4.1.4:** Для экспорта в Word должна использоваться библиотека python-docx.

**Требование 4.1.5:** Запрещено использование сторонних библиотек, кроме явно указанных.

### 4.2 Ограничения разработки
**Требование 4.2.1:** Запрещено использование регулярных выражений.

**Требование 4.2.2:** Запрещено применение сложных объектно-ориентированных конструкций - допускается только один основной класс или вообще без классов.

**Требование 4.2.3:** Все надписи в интерфейсе должны быть на русском языке.

**Требование 4.2.4:** Весь код должен быть прокомментирован на русском языке.

### 4.3 Требования к удобству использования
**Требование 4.3.1:** Интерфейс должен быть интуитивно понятен пользователю с минимальными компьютерными навыками.

**Требование 4.3.2:** Время обучения работе с программой не должно превышать 15 минут.

**Требование 4.3.3:** Программа должна содержать информативные сообщения об ошибках и успешных операциях.

### 4.4 Требования к надежности
**Требование 4.4.1:** Программа должна корректно обрабатывать некорректный ввод данных (пустые поля, неверный формат даты).

**Требование 4.4.2:** При возникновении ошибок программа не должна завершаться аварийно, а предоставлять пользователю информацию о проблеме.

**Требование 4.4.3:** Данные должны сохраняться автоматически при каждом добавлении/удалении записи.

## 5. Бизнес-процессы

### 5.1 Основной процесс: Добавление достижения
```
Шаг 1: Пользователь открывает программу
Шаг 2: Переходит на вкладку "Добавить достижение"
Шаг 3: Заполняет поля формы:
  - Вводит название достижения
  - Указывает дату в формате ГГГГ-ММ-ДД
  - Выбирает тип из выпадающего списка
  - Выбирает уровень из выпадающего списка
  - Вводит описание (опционально)
Шаг 4: Нажимает кнопку "Сохранить достижение"
Шаг 5: Система проверяет обязательные поля
Шаг 6: Система сохраняет данные в БД
Шаг 7: Система очищает форму и показывает сообщение об успехе
```

### 5.2 Процесс: Формирование отчета
```
Шаг 1: Пользователь переходит на вкладку "Мои достижения"
Шаг 2: Просматривает список своих достижений
Шаг 3: Нажимает кнопку "Экспорт в Word"
Шаг 4: Система формирует документ Word со всеми достижениями
Шаг 5: Система сохраняет файл в папке с программой
Шаг 6: Система показывает сообщение об успешном экспорте
```

### 5.3 Процесс: Удаление достижения
```
Шаг 1: Пользователь выбирает достижение в списке
Шаг 2: Нажимает кнопку "Удалить выбранное"
Шаг 3: Система запрашивает подтверждение удаления
Шаг 4: После подтверждения система удаляет запись из БД
Шаг 5: Система обновляет список достижений
Шаг 6: Система показывает сообщение об успешном удалении
```

## 6. Архитектурные требования

### 6.1 Структура проекта
**Требование 6.1.1:** Все файлы проекта должны находиться в одной папке:
- `main.py` - основной исполняемый файл
- `types.json` - файл конфигурации с типами достижений
- `достижения.db` - файл базы данных (создается автоматически)
- `достижения_ГГГГ-ММ-ДД.docx` - файлы отчетов (создаются при экспорте)

**Требование 6.1.2:** Файл `types.json` должен быть в кодировке UTF-8 и содержать валидный JSON-массив строк.

### 6.2 Требования к интерфейсу
**Требование 6.2.1:** Главное окно должно содержать две вкладки:
- "Добавить достижение" - форма для ввода новых данных
- "Мои достижения" - список сохраненных достижений и управление ими

**Требование 6.2.2:** Интерфейс должен быть адаптивным и корректно отображаться при разрешении экрана от 800x600 пикселей.

## 7. Бизнес-выгоды

### 7.1 Количественные выгоды
1. **Экономия времени** - сокращение времени на ведение учета достижений с 5 часов в месяц до 30 минут
2. **Повышение эффективности** - увеличение количества правильно оформленных заявок на 50%
3. **Снижение рисков** - устранение потерь информации об достижениях

### 7.2 Качественные выгоды
1. **Профессиональный имидж** - возможность быстро сформировать профессиональное портфолио
2. **Конкурентное преимущество** - систематизированная информация о достижениях дает преимущество при поступлении/трудоустройстве
3. **Личная организация** - развитие навыков систематизации и анализа собственных успехов
4. **Мотивация** - визуализация достижений повышает мотивацию к дальнейшему развитию

### 7.3 Стратегические выгоды
1. **Стандартизация подхода** - создание единого формата учета достижений в образовательном учреждении
2. **Аналитика развития** - возможность анализа динамики учебных успехов
3. **Интеграционный потенциал** - возможность интеграции с другими системами (электронный дневник, система управления обучением)

## 8. Ограничения и допущения

### 8.1 Ограничения
1. Программа предназначена для индивидуального использования
2. Нет возможности сетевого доступа к данным
3. Нет функции печати напрямую из программы
4. Нет возможности прикрепления файлов (скан сертификатов)
5. Нет функции поиска по содержимому

### 8.2 Допущения
1. Пользователь имеет базовые навыки работы с компьютером
2. На компьютере установлен Microsoft Word или совместимый редактор для просмотра отчетов
3. Пользователь понимает важность регулярного экспорта данных для резервного копирования
4. Программа устанавливается на компьютеры под управлением Windows, но совместима с Linux и macOS
