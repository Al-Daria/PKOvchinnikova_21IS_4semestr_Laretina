## 1. ОБЩИЕ СВЕДЕНИЯ

### 1.1. Назначение документа
Настоящий документ определяет план тестирования программного комплекса "Project Manager" версии 1.0. План описывает цели, объем, стратегию, ресурсы и расписание проведения тестирования.

Документ применяется для организации, планирования и контроля процесса тестирования системы управления проектами.

## 4. СТРАТЕГИЯ ТЕСТИРОВАНИЯ

### 4.1. Типы тестирования

#### 4.1.1. Функциональное тестирование
- **Цель**: Проверка соответствия функциональным требованиям
- **Методы**: Черный ящик, эквивалентное разбиение, анализ граничных значений
- **Инструменты**: pytest, unittest

#### 4.1.2. Интеграционное тестирование
- **Цель**: Проверка взаимодействия между модулями
- **Методы**: Нисходящая, восходящая интеграция
- **Инструменты**: pytest, Docker

#### 4.1.3. Системное тестирование
- **Цель**: Проверка системы в целом
- **Методы**: End-to-end тестирование
- **Инструменты**: Автоматизированные скрипты

#### 4.1.4. Регрессионное тестирование
- **Цель**: Проверка отсутствия регрессий после изменений
- **Методы**: Повторное выполнение ключевых тестов
- **Инструменты**: Автоматизированные тесты

### 4.2. Подходы к тестированию

| Подход | Применение | Описание |
|--------|------------|-----------|
| Ручное тестирование | Исследовательское тестирование, тестирование UI | Выполнение тестов вручную |
| Автоматизированное тестирование | Регрессионное, модульное, интеграционное | Использование автоматизированных скриптов |
| Статическое тестирование | Анализ кода, рецензирование | Проверка без выполнения кода |

## 8. ПРОЦЕСС УПРАВЛЕНИЯ ДЕФЕКТАМИ

### 8.1. Классификация дефектов

| Уровень | Описание | Время реакции | Примеры |
|---------|-----------|---------------|----------|
| Критический | Блокирует работу системы | 2 часа | Система не запускается, потеря данных |
| Высокий | Существенно влияет на функциональность | 8 часов | Основная функция не работает |
| Средний | Частично влияет на функциональность | 24 часа | Функция работает с ограничениями |
| Низкий | Косметические проблемы | 48 часов | Опечатки в интерфейсе |

### 8.2. Жизненный цикл дефекта

```
НОВЫЙ → НАЗНАЧЕН → В РАБОТЕ → ИСПРАВЛЕН → ПРОВЕРЕН → ЗАКРЫТ
       ↑                           ↓
       └────── ОТКЛОНЕН ←──────────┘
```

### 8.3. Шаблон описания дефекта

```markdown
ИДЕНТИФИКАТОР: [PROJ-BUG-XXX]
ЗАГОЛОВОК: [Краткое описание]
ПРИОРИТЕТ: [Critical/High/Medium/Low]
МОДУЛЬ: [Название модуля]
ШАГИ ВОСПРОИЗВЕДЕНИЯ:
1. [Шаг 1]
2. [Шаг 2]
3. [Шаг 3]
ОЖИДАЕМЫЙ РЕЗУЛЬТАТ: [Что должно было произойти]
ФАКТИЧЕСКИЙ РЕЗУЛЬТАТ: [Что произошло фактически]
ОКРУЖЕНИЕ: [ОС, версия, браузер и т.д.]
ВЛОЖЕНИЯ: [Скриншоты, логи, видео]
```

## 9. МЕТРИКИ И ОТЧЕТНОСТЬ

### 9.1. Ключевые метрики

| Метрика | Формула расчета | Целевое значение |
|---------|-----------------|------------------|
| Процент успешных тестов | (Успешные тесты / Всего тестов) × 100% | ≥95% |
| Эффективность тестирования | (Найденные дефекты / Всего дефектов) × 100% | ≥90% |
| Плотность дефектов | Количество дефектов / КЛП | ≤0.1 |
| Время устранения дефектов | Среднее время исправления дефекта | ≤8 часов |

### 9.2. Отчетные документы

| Отчет | Частота | Содержание | Получатели |
|-------|---------|------------|------------|
| Ежедневный статус | Ежедневно | Прогресс, проблемы, планы | Руководство проекта |
| Отчет о дефектах | По мере обнаружения | Новые и исправленные дефекты | Команда разработки |
| Итоговый отчет | По завершении | Результаты, рекомендации | Заинтересованные стороны |
